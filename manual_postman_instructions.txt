### üîç Individual test cases (executed in Postman)

1. **Downstream ‚Äì missing `user` field**
   - I create a POST to `http://localhost:8085/login`.
   - I add the `missingUser` JSON payload.
   - I expect a **400** response; the body should say something like
     `"Missing 'user' key"`.

2. **Downstream ‚Äì valid request**
   - POST to `http://localhost:8085/login` with `validDownstream` payload.
   - I should get **200** and a JSON body containing `user: "bob"`, a
     `token` value, and a `timestamp`.

3. **Proxy ‚Äì request without user**
   - POST to `http://localhost:8000/login` with `missingUser`.
   - The proxy should immediately return **400** (no call to downstream).

4. **Proxy ‚Äì normal forwarding**
   - POST to `http://localhost:8000/login` with `validProxy`.
   - I expect **200**.  The returned JSON must **not** have a `user` field (the
     proxy strips it), but it should include other data such as `token` and
     `timestamp` from the downstream service.

5. **Proxy ‚Äì downstream omits user**
   - POST to `http://localhost:8000/login-no-user` with `userOnly`.
   - I should see **400** because the proxy detects the missing `user` in the
     downstream response and treats it as an error.

6. **Proxy ‚Äì downstream returns non‚ÄëJSON**
   - POST to `http://localhost:8000/login-invalid` with `userOnly`.
   - The proxy will try to parse the reply as JSON and fail, giving me a **500**
     error.

7. **Proxy ‚Äì client sends invalid JSON**
   - POST to `http://localhost:8000/login` but set `Content-Type` to
     `text/plain` and use the raw text `this is not json`.
   - I expect a **400** response (the proxy rejects the invalid JSON before
     forwarding).
